Introduction & Motivation

Recursive queries are fundamental for analyzing complex graph relationships in critical applications:

  • Genealogy: Ancestry tracing across multiple generations
  • Security: Role-based access control (RBAC systems)
  • Finance: Transaction chains and fraud detection
  • Networks: Reachability and path analysis

Research Scope

7 database systems evaluated across 12 graph topologies. Standard benchmarks (TPC-H, TPC-DS) lack recursive workloads, creating a critical gap in performance evaluation.

3 Recursion Variants

  • Left: path(X,Z), edge(Z,Y)
  • Right: edge(X,Z), path(Z,Y)
  • Non-Linear: path(X,Z), path(Z,Y)

Evaluation: Graph sizes 100-10,000 nodes • 5 runs per configuration

Right Recursion (SQL)
-- Base case: direct edges
WITH RECURSIVE tc AS (
  SELECT x, y FROM edge
  
  UNION
  
  -- Recursive: extend paths
  SELECT edge.x, tc.y
  FROM edge
  JOIN tc ON edge.y = tc.x
)
SELECT COUNT(*) FROM tc;
Right Recursion (Prolog) 
path(X, Y) :- edge(X, Y).
path(X, Y) :- 
  edge(X, Z), 
  path(Z, Y).

Architectural Limitations Discovered

SingleStore (Excluded)

Issue: "Crippled CTEs" — explicitly disallows:

  • GROUP BY in recursive branch
  • Aggregate functions (COUNT, SUM)
  • UNION (only UNION ALL)

Documentation: v8.7 (confirmed 2024)

MongoDB (Performance Issues)

$graphLookup: Stage memory limitations

Struggles with dense graphs (100-node graph took >1,500s) due to intermediate result explosion and 100MB stage memory cap

Neo4j (Scalability Challenge)

Cypher Path Enumeration: Pre-deduplication issues

Complete graphs cause path explosion before deduplication can occur

Finding 1: Dense Graph Performance

Complete & Max Acyclic Graphs

On dense graphs (edge density > 0.5), different execution strategies show significant performance variance.

  • DuckDB: Vectorized execution benefits from batch processing
  • XSB Prolog: Tabled resolution handles high edge density efficiently
  • PostgreSQL/MariaDB: Row-based iteration overhead evident
Key Insight:

Architectural differences amplify on dense graphs with \(O(n^2)\) edges

Finding 2: Sparse Graph Performance

Path & Cycle Graphs

On sparse graphs (edge density < 0.01), different factors dominate performance.

  • XSB Prolog: Goal-directed evaluation minimizes redundant work
  • SQL Systems: Iteration overhead becomes more prominent
  • Graph Structure:

    Linear topology (\(O(n)\) edges) changes trade-offs

Key Insight: Sparse graphs favor systems with lower per-iteration overhead

Critical Finding: Correctness Failures

DuckDB Non-Linear Recursion Bug

Incorrect results on multiple graph topologies (v0.9.2)

Root Cause: While DuckDB accepts non-linear recursion syntax, it fails to properly compute transitive closure when paths require composition of two derived paths.

✓ Correct (Prolog) 
path(X, Y) :- edge(X, Y).
path(X, Y) :- 
  path(X, Z), 
  path(Z, Y).

-- Properly handles
-- transitive composition
✗ Buggy (DuckDB) 
WITH RECURSIVE tc AS (
  SELECT x, y FROM edge
  UNION
  SELECT tc1.x, tc2.y
  FROM tc AS tc1, tc AS tc2
  WHERE tc1.y = tc2.x
)
-- Misses derived→derived paths

Safe Alternatives

  • ✓ PostgreSQL: Rejects non-linear syntax
  • ✓ CockroachDB: Rejects non-linear syntax
  • ✓ MariaDB: Explicit flag required
  • ✓ XSB Prolog: Tabling handles correctly

Impact

Affected graphs: Grid, Balanced Tree, and other topologies
Workaround: Use linear recursion variants
Status: Reported to DuckDB team

Performance Comparison Summary

Key Observations: Performance varies significantly based on graph structure, with no single system dominating across all topologies.
