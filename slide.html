Introduction & Motivation

Recursive queries are fundamental for analyzing complex graph relationships in critical applications:

  • Genealogy: Ancestry tracing across 10+ generations.
  • Security: Multi-level access control (RBAC systems).
  • Finance: Money laundering detection chains.

Research Gap

7 database systems tested across 12 graph topologies. Standard benchmarks (TPC-H, TPC-DS) lack recursive workloads, creating a blind spot in performance evaluation.

12 Graph Topologies

  • Dense: Complete, Max Acyclic
  • Sparse: Path, Cycle, Binary Tree
  • Structured: Grid (2D, 3D), Balanced Tree

3 Recursion Variants

  • Left: path(X,Z), edge(Z,Y)
  • Right: edge(X,Z), path(Z,Y)
  • Non-Linear: path(X,Z), path(Z,Y)

Evaluation: Sizes 100-10,000 nodes • 5 runs per config

Right Recursion (SQL)
-- Base case: direct edges
WITH RECURSIVE tc AS (
  SELECT x, y FROM edge
  
  UNION
  
  -- Recursive: extend paths
  SELECT edge.x, tc.y
  FROM edge
  JOIN tc ON edge.y = tc.x
)
SELECT COUNT(*) FROM tc;
Right Recursion (Prolog) 
path(X, Y) :- edge(X, Y).
path(X, Y) :- 
  edge(X, Z), 
  path(Z, Y).

Architectural Limitations Discovered

SingleStore (Excluded)

Issue: "Crippled CTEs" — explicitly disallows:

  • GROUP BY in recursive branch
  • Aggregate functions (COUNT, SUM)
  • UNION (only UNION ALL)

Documentation: v8.7 (confirmed 2024)

MongoDB (Non-Viable)

$graphLookup: 100MB stage memory cap

1,500s for 100-node complete graph

~150x slower than PostgreSQL baseline

Neo4j (Scalability Issue)

Cypher Path Enumeration: Pre-deduplication explosion

Example: 1,000-node complete graph → O(n!) intermediate paths

Timeout threshold: 3,600s (1 hour)

Finding 1: Dense Graphs (Vectorization Wins)

DuckDB Dominance

On Complete and Max Acyclic graphs (edge density > 0.5), columnar processing excels.

  • SIMD Operations: Process 8-16 tuples/cycle (AVX-512)
  • Cache Efficiency: 95%+ L1/L2 hit rate on batch joins
  • vs. XSB: Competitive but hampered by per-tuple overhead
Bottleneck Shift: From iteration logic → join throughput
10x Faster than PostgreSQL (10K nodes)
3.2x Faster than XSB Prolog

Finding 2: Sparse Graphs (Tabling Wins)

XSB Prolog Efficiency

On Path and Cycle graphs (edge density < 0.01), iteration overhead dominates.

  • SLG Resolution: Memoization avoids redundant sub-goals
  • Goal-Directed: Explores only reachable nodes
  • DuckDB Overhead: Vector setup cost > tuple processing
Example: 10,000-node path graph → XSB: 2.3s, DuckDB: 4.1s

Iteration Metrics

XSB: 50µs/iteration
DuckDB: 180µs/iteration
PostgreSQL: 220µs/iteration

Measured on 10K-node path graph (AMD EPYC 7763)

Critical Finding: Correctness Failures

DuckDB Non-Linear Recursion Bug

Incorrect results for 7 of 12 graph structures (v0.9.2)

Root Cause: While DuckDB accepts non-linear recursion syntax, it fails to discover paths requiring composition of two derived paths.

✓ Correct (Prolog) 
path(X, Y) :- edge(X, Y).
path(X, Y) :- 
  path(X, Z), 
  path(Z, Y).

-- Properly handles transitive composition
✗ Buggy (DuckDB) 
WITH RECURSIVE tc AS (
  SELECT x, y FROM edge
  UNION
  SELECT tc1.x, tc2.y
  FROM tc AS tc1, tc AS tc2
  WHERE tc1.y = tc2.x
)
-- Misses derived→derived paths

Safe Alternatives

  • ✓ PostgreSQL: Rejects non-linear syntax
  • ✓ CockroachDB: Rejects non-linear syntax
  • ✓ MariaDB: Explicit flag required
  • ✓ XSB Prolog: Tabling handles correctly

Impact

Missing paths: 15-40% on affected topologies
Affected graphs: Grid, Balanced Tree, PA variants
Status: Reported (Issue #13038)
